# 邀请链接与房间码设计

> 状态：已实现，待测试服验收  
> 产品：德扑道场（Poker Dojo）  
> 更新时间：2026-07-23

## 实现记录

2026-07-23 已按本文档完成第一版实现：

- 服务端生成高强度 `inviteToken` 和唯一四位 `joinCode`；
- 增加链接/房间码授权、每账号与每 IP 限速；
- `join_room` 不再接受 `byCode` 授权，已授权状态以服务器 `authorized` 为准；
- 房主可查看、复制、锁定/开放及重置邀请；
- 客户端支持 fragment 捕获、登录/注册后的邀请续接，并同时使用页面内存与
  `sessionStorage` 兼容不同 WebView；
- 大厅改为四位房间码，内部 `roomId` 不再展示为邀请凭证；
- 已通过本地双账号浏览器回归，尚未部署测试服。

## 1. 背景

当前加入房间的方式是：

- 服务器创建一个 6 位数字 `roomId`；
- 玩家从大厅列表进入时只能观战；
- 玩家在大厅手动输入 `roomId`，客户端发送
  `join_room({ roomId, byCode: true })` 后获得下场资格；
- 已在座、站起或曾获得资格的玩家可以重新进入。

这套流程有两个问题：

1. 玩家需要在聊天软件和游戏之间记忆、复制、输入 6 位数字，不符合熟人局的自然邀请习惯；
2. `byCode` 是客户端声明，服务器没有验证独立的秘密凭证。大厅数据又包含
   `roomId`，懂得构造 Socket 消息的用户可以绕过界面获得资格。因此当前机制只是
   UI 门槛，不是真正的服务器授权。

## 2. 目标

本功能提供互补的两种邀请方式：

1. **复制邀请链接**：适合同一台手机或电脑接收消息后直接点击；
2. **输入四位房间码**：适合手机收到消息、但玩家想在未登录聊天软件的电脑上游戏。

两种方式最终都由服务器验证，并把登录用户加入房间的 `authorized` 授权集合。

同时满足：

- 未获得邀请的用户从大厅只能观战；
- 已获授权的用户退出、掉线、换设备后仍可重新进入；
- 房主可以锁定新玩家入场；
- 房主可以在邀请凭证泄露后重置邀请；
- 不改变 SNG 和现金桌现有的入座、开赛与重连规则。

## 3. 非目标

第一版不包含：

- 房主自由填写自定义房间码；
- 邀请审批队列或逐人批准；
- 永久房间或永久邀请链接；
- 短信、邮件、微信等平台内的直接分享接口；
- iOS Universal Links / Android App Links（第一版链接在浏览器中打开）；
- 二维码生成（可作为后续增强）；
- 将房间状态从内存迁移到数据库。

## 4. 核心产品决策

### 4.1 一个邀请系统，两种入口

每个房间创建时自动生成：

```text
邀请链接：https://pokerdojo.space/#/join/<inviteToken>
房间码：7284
```

`inviteToken` 和 `joinCode` 都只用于证明“用户拿到了房主分享的邀请”，验证成功后的权限完全相同。

### 4.2 房间码采用四位随机数字

- 格式固定为 `0000`～`9999`，必须保留前导零；
- 使用 Node.js `crypto.randomInt(10000)` 生成，不使用 `Math.random()`；
- 在当前所有活跃房间中唯一；
- 房间结束、解散或被清理后，该号码可以被未来的新房间复用；
- 第一版不允许房主自由填写，但房主可以通过“重置邀请”获得新的随机码。

选择随机生成而不是自定义，是为了避免大量 `1111`、`8888`、生日等弱码，以及重名、字符规范和敏感内容问题。若后续确有需求，可以另行设计 4～8 位自定义字母数字码。

四位数字不是高强度密码。它的安全性依赖于：

- 玩家必须先登录；
- 房间生命周期有限；
- 服务端错误尝试限速；
- 房主可锁定入场和重置邀请；
- 高强度邀请链接作为主要邀请方式。

### 4.3 邀请链接使用高强度随机令牌

- `inviteToken` 使用 `crypto.randomBytes(16)` 或更高强度生成；
- 编码为 URL 安全的 Base64，不包含需要转义的字符；
- 不从 `roomId`、房间码、用户名或时间戳推导；
- 不写入访问日志、游戏日志或牌谱；
- URL 使用 fragment：

```text
https://pokerdojo.space/#/join/<inviteToken>
```

fragment 默认不会随 HTTP 请求发送给 Caddy/Express，可减少令牌进入访问日志和
Referer 的风险。客户端读取后通过已加密的 Socket.IO/WSS 连接提交给服务器验证。

### 4.4 `roomId` 降级为内部标识

现有 6 位 `roomId` 继续作为：

- `roomGames` 的键；
- Socket.IO room 名称；
- 内部事件和牌谱中的房间标识；
- 大厅列表进入观战时的目标标识。

但它不再是入场秘密，也不能单独授予下场资格。

## 5. 数据模型

每个 `roomGames[roomId]` 增加：

```js
{
    // 现有字段……
    authorized: new Set([ownerUserId]),

    invite: {
        token: "高强度 URL-safe 随机字符串",
        joinCode: "四位字符串，如 0316",
        entryLocked: false,
        version: 1,
        createdAt: 1780000000000
    }
}
```

说明：

- `authorized` 保存已经验证成功的 `userId`，房主创建时自动加入；
- `joinCode` 必须是字符串，防止 `0316` 变成 `316`；
- `entryLocked` 只禁止授权新的用户，不影响已授权用户重连；
- `version` 每次重置邀请时递增，便于调试和避免旧异步结果误用；
- 房间目前存储在内存，因此邀请信息也暂存内存，随房间一起销毁。

服务端应提供统一函数：

```js
authorizeWithInvite(game, user, credential)
```

所有邀请入口必须经过同一套校验，禁止分别实现出不同权限语义。

## 6. 授权模型

### 6.1 可以获得下场资格的情况

用户满足任意一项即可成为已授权成员：

- 是房主；
- 提交了当前有效的 `inviteToken`；
- 提交了当前有效的 `joinCode`；
- 已在 `game.authorized` 中；
- 已经是该房间的在座玩家或 `vacatedPlayers` 成员。

### 6.2 不能获得下场资格的情况

以下行为只能观战或收到错误提示：

- 仅从大厅列表点击进入；
- 仅知道 `roomId`；
- 客户端自行声明 `byCode: true`；
- 使用已经重置的旧链接或旧房间码；
- 房间已经锁定，且该用户此前没有获得授权；
- SNG 已经开始、满员或不满足现有入座条件。

### 6.3 锁定入场

房主开启 `entryLocked` 后：

- `authorized` 中已有的用户仍可重连、重新进入和按现有规则入座；
- 新用户即使持有正确链接或房间码也不能获得授权；
- 大厅观战规则保持不变；
- 房主可以再次开放入场。

锁定只控制“新授权”，不应把已经授权的朋友踢出房间。

### 6.4 重置邀请

房主点击“重置邀请”后：

- 同时重新生成 `inviteToken` 和 `joinCode`；
- 旧链接与旧房间码立即失效；
- `version += 1`；
- 已在 `authorized` 中的用户保留资格；
- `entryLocked` 状态保持不变；
- 操作需二次确认，避免误触导致尚未加入的朋友拿着旧邀请失败。

重置的用途是处理邀请被错误转发或泄露，而不是清空现有成员。

## 7. Socket 协议

事件名沿用项目的 snake_case 约定。

### 7.1 通过房间码加入

客户端：

```js
socket.emit('join_by_code', { code: '7284' });
```

服务端：

1. 校验格式必须为四位数字；
2. 执行限速；
3. 在活跃房间中查找匹配的 `joinCode`；
4. 校验房间存在、未结束、允许新授权；
5. 将用户加入 `authorized`；
6. 按房间类型调用统一的进入/入座流程。

失败时返回通用提示：

```text
房间码无效或当前不可加入
```

不要区分“码不存在”“房间已锁定”“房间刚结束”，避免帮助批量枚举活跃房间。

### 7.2 通过邀请链接加入

客户端：

```js
socket.emit('join_by_invite', { token: '<inviteToken>' });
```

服务端校验令牌、锁定状态和房间状态，成功后执行与房间码相同的授权与进入流程。

失败提示：

```text
邀请已失效或当前不可加入
```

### 7.3 获取邀请信息

房主进入房间或打开“邀请朋友”面板时：

```js
socket.emit('get_room_invite');
```

仅当前房间房主可收到：

```js
socket.emit('room_invite_info', {
    joinCode: '7284',
    inviteUrl: 'https://pokerdojo.space/#/join/...',
    entryLocked: false,
    version: 1
});
```

`inviteToken`、完整链接和 `joinCode` 不应放进广播的 `game_state`、公共房间列表或观战者可见数据。

### 7.4 房主管理事件

```js
socket.emit('set_entry_locked', { locked: true });
socket.emit('reset_room_invite');
```

两个事件都必须在服务器端验证：

- 当前用户是房主；
- Socket 当前确实位于目标房间；
- 房间仍然存在。

成功后仅向房主返回新的 `room_invite_info`，同时可以向房间广播不含秘密的状态提示，例如“房主已锁定入场”。

### 7.5 `join_room` 的调整

保留 `join_room({ roomId })` 用于：

- 大厅进入观战；
- 已授权成员重新进入；
- 断线重连。

删除 `byCode` 的授权语义。服务端必须忽略或拒绝客户端传入的 `byCode`，不能因为该布尔值授予资格。

为兼容部署期间可能仍打开着旧页面的客户端，可以暂时忽略 `byCode` 并按未授权观战处理；生产稳定后删除相关客户端代码。

## 8. 不同房型的成功加入行为

邀请验证只负责授予资格，后续仍复用现有房型规则。

### SNG

- 等待状态且未满员：验证成功后按现有逻辑入座；
- 已满、已开赛或已结束：不新增参赛者，提示不可加入；
- 已授权的原成员：允许按现有重连规则回来；
- 是否允许失败后转为观战，由客户端明确提示并让用户选择，第一版不要静默进入造成误解。

### 现金桌 / 训练赛

- 验证成功后以有下场资格的观众身份进入；
- 玩家点击空座位，再选择带入筹码并坐下；
- 已有座位、站起或留座的玩家按现有恢复逻辑处理；
- 房间结束后凭证随房间销毁。

## 9. 客户端流程

### 9.1 创建房间

创建成功进入牌桌后，房主自动看到一次邀请面板：

```text
邀请朋友加入

房间码
7284                    [复制房间码]

邀请链接
pokerdojo.space/…       [复制邀请链接]

手机收到消息可直接点击；
跨设备时可输入四位房间码。

[开放入场 / 锁定入场]  [重置邀请]
```

牌桌菜单增加“邀请朋友”，房主之后可以再次打开。

第一版只给房主显示邀请秘密。普通已授权玩家可以重新进入，但不在界面中展示完整邀请凭证。

### 9.2 输入房间码

大厅“加入比赛”改为：

```text
输入四位房间码
[ 7 ][ 2 ][ 8 ][ 4 ]  [加入]
```

实现要求：

- `inputmode="numeric"`；
- `maxlength="4"`；
- 保留前导零；
- 粘贴时过滤空格和非数字；
- 输入满四位后可以允许按 Enter 提交；
- 按钮提交期间防止重复点击；
- 文案不再称其为“房间号”。

### 9.3 打开邀请链接

页面启动时检查：

```text
#/join/<inviteToken>
```

处理顺序：

1. 验证 token 只包含允许的 URL-safe 字符且长度合理；
2. 暂存在 `sessionStorage.pendingInviteToken`；
3. 使用 `history.replaceState` 清除地址栏中的 fragment；
4. 如果未登录，正常显示登录/注册页；
5. 登录成功、Socket 鉴权连接完成后，自动发送 `join_by_invite`；
6. 成功或明确失败后删除 `pendingInviteToken`。

使用 `sessionStorage` 而不是 `localStorage`，避免邀请令牌长期残留。注册、邮箱验证和忘记密码流程切换页面时不能丢失 pending invite。

### 9.4 复制行为

优先使用 `navigator.clipboard.writeText()`；不可用时使用兼容回退方案。成功后给出短暂 toast：

```text
邀请链接已复制
房间码 7284 已复制
```

不要把完整 token 输出到普通页面日志或错误上报中。

## 10. 防枚举与滥用

### 10.1 房间码限速

第一版至少实现两层内存限速：

- 每账号：60 秒内最多 5 次失败；
- 每 IP：10 分钟内最多 20 次失败。

超过限制后进入冷却，并返回与普通失败不同但不泄露房间信息的提示：

```text
尝试次数过多，请稍后再试
```

成功加入后可以清除该账号的失败计数，但不必清除整个 IP 的计数。限速数据只需保存在内存，并定期清理过期记录。

### 10.2 其他约束

- 只有已登录、Socket 鉴权成功的用户可以提交邀请；
- 对 token 和 code 先做长度/字符校验，再查询房间；
- 所有权限判断以服务器状态为准；
- 不在房间列表、`game_state`、聊天消息、牌谱或服务端普通日志中广播邀请凭证；
- 邀请校验失败不调用 `authorize()`；
- 一个账号已在其他房间时，沿用现有离开/切房规则，不因邀请绕过结算或座位处理。

## 11. 大厅与观战

保留当前“公开列表可观战”的产品逻辑：

- 大厅房间卡片仍可携带内部 `roomId`，用于点击观战；
- 未授权用户点击房间卡片，只能成为 spectator；
- 观战者看不到房间码和邀请链接，也不能坐下；
- 已授权用户的房间卡片显示“重新加入”；
- 公开大厅是否未来支持“隐藏房间”不在第一版范围内。

注意：公开 `roomId` 本身不再构成安全问题，因为服务器不再接受
`byCode: true` 作为授权证明。

## 12. 生命周期与异常处理

- 房间创建：生成唯一 `joinCode` 和 `inviteToken`；
- 房间重置邀请：原子替换两者；
- 房间锁定：仅改变是否接受新授权；
- 房间解散、比赛结束或空房清理：邀请凭证随 `roomGames` 一起销毁；
- 服务器重启：当前内存房间本来就会消失，旧邀请自然失效；
- 掉线重连：依赖 `authorized`、在座玩家和 `vacatedPlayers`，不要求再次提交邀请；
- 多标签页/重复点击：`authorized` 使用 `Set`，授权必须幂等，不能重复扣金币或重复入座；
- 邀请验证成功但入座失败：授权可以保留，但扣金币、占座等后续操作必须保持现有原子性。

## 13. 兼容与迁移

本功能不迁移历史数据，因为房间状态目前只存在内存。

上线时需要同时更新服务端与 `index.html`：

1. 服务端先增加新凭证和事件；
2. 删除 `byCode` 的授权能力；
3. 客户端把原“房间号”输入框替换为四位房间码；
4. 客户端增加链接解析与邀请面板；
5. 部署会重启服务，旧内存房间和旧 6 位邀请方式直接结束；
6. 用户账号、金币、牌谱等持久数据不受影响。

不需要保留旧 6 位房间号加入方式。保留它反而会留下绕过新授权模型的后门。

## 14. 实现建议

建议按以下顺序实现：

1. 抽取统一的“进入房间”和“已授权判断”函数，避免三种入口复制逻辑；
2. 增加安全随机 token、四位唯一 code 和房间邀请状态；
3. 实现 `join_by_code`、`join_by_invite` 及限速；
4. 移除 `byCode` 授权；
5. 实现房主查看、锁定和重置事件；
6. 实现客户端 pending invite 登录续接；
7. 实现邀请面板、复制按钮和四位码输入；
8. 补齐自动化测试，再部署测试服验收。

不要在改造过程中改变金币兑换、SNG 报名费、现金桌买入或断线重连规则。

## 15. 测试清单

### 服务端单元/Socket 测试

- 创建两个房间时，四位码和 token 都不同；
- 强制制造 code 冲突时会重新生成；
- `0316` 能保留前导零并正确匹配；
- 正确 code 和 token 都能把用户加入 `authorized`；
- 错误 code/token 不授权；
- 只知道公开 `roomId` 并发送 `byCode: true` 不能获得资格；
- 大厅点击仍只能观战；
- 未授权现金桌观战者不能坐下；
- 已授权现金桌玩家可以选座和买入；
- SNG 等待时邀请成功可入座，开赛后新用户不可加入；
- 已授权用户在锁定后仍可重连，新用户不能加入；
- 重置后旧 code/token 失效，新凭证有效；
- 重置不会清空既有 `authorized`；
- 非房主不能查看、锁定或重置邀请；
- 重复提交同一邀请不会重复入座或重复扣金币；
- 每账号和每 IP 限速生效；
- 公共 `room_list` 和 `game_state` 不含 token、code。

### 客户端测试

- 已登录用户点击链接后直接进入；
- 未登录用户点击链接，登录后自动继续进入；
- 新用户走完注册和邮箱验证码后仍能继续邀请；
- pending token 处理后从地址栏和 `sessionStorage` 清除；
- 手机可以复制并打开邀请链接；
- 电脑输入四位码可以加入；
- 前导零房间码可以输入、粘贴和提交；
- 复制失败时有兼容回退和明确提示；
- SNG 已满/已开赛、房间锁定、邀请失效时提示准确；
- 房主锁定和重置后面板状态立即更新。

## 16. 验收标准

满足以下条件即可认为第一版完成：

1. 房主创建房间后无需手动组织房间号，可以一键复制邀请链接；
2. 朋友在同设备点击链接，登录后可直接进入；
3. 跨设备玩家只需输入四位数字即可进入；
4. 公开大厅用户仍只能观战，无法伪造 `byCode` 下场；
5. 房主可以锁定新入场并重置泄露的邀请；
6. 两种邀请方式不会破坏 SNG、现金桌、金币、座位和重连现有行为；
7. 邀请秘密不出现在公共广播和常规日志中；
8. 自动化测试覆盖授权绕过、限速、锁定、重置和两种房型。
